Instrument accepted-vs-confirmed trait logging so field tests can prove whether structure guest creation is being rolled back after initial acceptance.

## Highlights
- Log per-state summaries for `GuestsTrait`, `UserAccessTrait`, and `BasicUserSchedulesSettingsTrait` using explicit `accepted` / `confirmed` labels.
- Preserve the existing merged trait summaries while adding state-specific counts, payload lengths, and key guest/user identifiers for debugging.
- Add parser unit coverage for rank-label formatting and for retaining both accepted and confirmed entries for the same trait.
- Record the `b26` field result as a transient accepted guest/schedule update that reverts before any durable guest, user-access, or pincode state appears.

## Why
- The `2026-03-15` `b26` run with guest name `Test1` briefly surfaced `Test1` in raw `GuestsTrait` and schedule chunks, then reverted to the prior state within seconds.
- The existing logs required manual reconstruction across many chunks to see that rollback pattern.
- This build makes the accepted/confirmed split explicit so the next field run can answer whether Nest is rolling back the create request or expecting another write step.
