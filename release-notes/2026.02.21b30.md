Disable the experimental structure-side guest-create service after field testing showed it can knock a Nest device offline.

## Highlights
- Hard-disable `experimental_create_guest` at the Home Assistant service layer so the unsafe command cannot be dispatched again by accident.
- Add a second safety gate in the API client so direct/internal calls also fail immediately with a clear error.
- Mark the service description as disabled pending investigation.
- Add unit coverage for the disabled experimental path.

## Why
- This is a safety rollback. The experimental `GuestsTrait.CreateGuestRequest` path no longer meets the bar for field testing after a call appears to have taken the Nest device offline.
- Until the outage mechanism is understood, preventing further dispatch is more important than collecting more response logs.

## Validation
- `./.venv/bin/python -m unittest tests/test_api_client.py tests/test_protobuf_handler.py tests/test_passcode_crypto.py tests/test_passcode_utils.py`
- `./.venv/bin/python -m compileall custom_components/nest_yale_lock tests/test_api_client.py tests/test_protobuf_handler.py`

## Breaking changes
- `experimental_create_guest` is intentionally disabled and now raises an error instead of sending a command.
