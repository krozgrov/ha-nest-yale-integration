Instrument accepted structure guest-create commands so field tests can distinguish duplicate-name no-ops from real guest or pincode changes.

## Highlights
- Warn when `experimental_create_guest` is called with a guest name that already exists on the structure.
- After an accepted structure command, poll state a few times instead of relying on one immediate refresh.
- Diff before/after structure guest metadata, guest user-access records, and structure/device pincode slot fingerprints.
- Extract lightweight resource/type hints from the raw command response and log them alongside the observed state delta.
- Add unit coverage for the new API-client helper logic.

## Why
- The first structure-side `experimental_create_guest` field run returned a successful command response, but the immediate observe refresh showed no new guest IDs.
- That result is ambiguous: it could be delayed propagation, duplicate-name idempotency, or a silent no-op.
- This build makes that ambiguity observable without changing the existing encrypted `set_guest_passcode` path.
