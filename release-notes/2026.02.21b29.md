Inspect top-level status details, trait events, and nested `Any` payloads in successful `SendCommandResponse` logs so field tests can pinpoint where guest-create rollback metadata is hiding.

## Highlights
- Extend successful `nestlabs.gateway.v1.SendCommandResponse` summaries to include top-level `google.rpc.Status.details`, `TraitOperation.event`, and recursively nested `Any` envelopes embedded inside successful command payloads.
- Surface nested response payload metadata without dumping raw protobuf blobs or sensitive passcode material.
- Add unit coverage for nested `Any` extraction and for event/detail logging on the success path.
- Fix a parser regression in accepted-vs-confirmed trait entry retention so separate `accepted` and `confirmed` snapshots for the same descriptor are both preserved locally.
- Record the `b28` field result more precisely: the gateway reports a completed `guests` operation for `TestMCC`, but still no durable guest, user-access, or pincode state change follows.

## Why
- `b28` proved the structured `traitOperations` view is still incomplete for this path: it only exposes the original `CreateGuestRequest`, even though raw-byte hints still show `CreateGuestResponse`.
- This build moves one level deeper into the successful command envelope so the next field run can tell whether Nest is returning rollback metadata in `Status.details`, `TraitOperation.event`, or another nested `Any` payload.
